## 발견한 디자인 이슈 (본 세션 미해결, 후속 ADR 후보)

**test-templates 가 매 push 마다 fresh scaffold + uv sync 로 외부 라이브러리 최신 호환 버전을 가져오는 구조 → 외부 stub 변동 (이번엔 typer 0.25.1) 이 사용자 push 를 직접 차단.**

근거:
- `pre-push → make verify → test-templates → scaffold cli → uv sync` 가 매번 fresh.
- `[[feedback-ci-cost-sensitive]]` 메모리: GHA 못 씀 → pre-push 가 게이트 책임. CI 분리 옵션은 비용 제약.
- typer 류는 동적 메타프로그래밍 + pyright strict 와 항상 충돌 가능 (이번엔 `reportUnknownMemberType`). 다음엔 다른 룰 또는 다른 라이브러리가 깰 수 있음.

후속 결정 후보 (ADR):
- **A**: 템플릿 별 lockfile (`uv.lock`, `yarn.lock`) 을 templates/ 에 commit 해 test-templates 에서 frozen install. 외부 변동 완전 차단. trade-off: 의존성 자동 보안 패치 늦어짐, `eol-check.sh` 와 별개 lock-refresh 워크플로 필요.
- **B**: pre-push 의 verify 범위 축소 (test-install + lint 만), test-templates 는 별도 cadence (nightly local cron, GitHub Actions Free 한도 절약 모드). trade-off: pre-push 후 머지된 깨진 템플릿 발견 지연.
- **C**: pyright config 의 외부 라이브러리 호환 룰 (이번 `reportUnknownMemberType` 같이) 을 사전 풀어두기 — 보수적, 본 세션 처방의 연장.

→ 본 세션은 C 의 즉시 처방만 (cli 한정). A 또는 B 는 별개 세션의 ADR 로 결정.
