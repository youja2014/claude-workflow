# Agentic Workflow Rules

Claude 를 에이전트로 활용해 대형/복잡한 작업을 빠르게 처리할 때의 규칙. 근거: Anthropic 공식 문서(`code.claude.com/sub-agents`, `claude-code-best-practices`, `multi-agent-research-system`) + 실측.

## 핵심 원칙 — 컨텍스트가 가장 비싼 자원
- 컨텍스트 윈도는 빠르게 차고, 차오를수록 성능이 떨어진다. "빠르게 큰 걸 만든다" = "컨텍스트를 아껴 쓴다".
- 거의 모든 안티패턴은 컨텍스트 오염에서 비롯된다.

## 컨텍스트 관리
- 무관한 작업으로 넘어갈 때 `/clear` 로 대화 초기화
- 긴 세션은 `/compact` 로 요약 압축
- 파일을 많이 읽는 탐색/리서치는 **서브에이전트에 위임** — 별도 컨텍스트에서 돌고 요약만 회신
- 항상 로드되는 컨텍스트(CLAUDE.md, common 룰)는 최소화 — 상세는 경로 스코프 룰/skill 온디맨드로

## 작업 워크플로 — Explore → Plan → Code → Commit
- 큰 기능은 탐색(plan mode: 읽기만, 변경 X) → 계획(`/plan`) → 구현 → 커밋 순서로 분리
- "엉뚱한 문제를 푸는 것"을 막는 게 목적
- 단순 작업(1-2줄 diff, rename, typo)은 이 단계를 건너뛴다

## 서브에이전트
- 각 서브에이전트는 **자기 컨텍스트에서 독립 실행 후 요약만 회신** — 메인 컨텍스트 보존이 핵심 목적
- **서브에이전트는 다른 서브에이전트를 못 띄운다 (깊이-1, 실측 확인)**. `tools:*` 라도 불가 → 오케스트레이션은 메인 대화에서 실행되는 **skill** 로
- 장문 결과를 그대로 받지 말 것 — 요약을 요구해 컨텍스트 오염 방지
- 모델 티어링: 단순 스캔/조회 `haiku`, 일반 코딩/리뷰 `sonnet`, 설계/난제 `opus`
- frontmatter `model:` 유효값 = `sonnet`/`opus`/`haiku`/풀 모델 ID/`inherit`(기본)

## 멀티에이전트 오케스트레이션 — 언제 쓰고 언제 안 쓰나
- **잘 맞음**: breadth-first 리서치, 단일 컨텍스트를 넘는 작업, 독립적 방향 병렬 조사
- **안 맞음**: 대부분의 코딩(병렬화 하위작업 적음), 공유 컨텍스트/강한 의존성이 필요한 작업
- 비용: 단일 채팅 1x · 에이전트 ~4x · 멀티에이전트 ~15x 토큰 → 복잡도가 비용을 정당화하는 고가치 작업에만
- 코딩 자동화는 멀티에이전트 fan-out 보다 **단일 에이전트 + 격리된 탐색 서브에이전트** 가 보통 낫다

## 노력 스케일링 (과투자 방지)
- 단순 사실 확인: 1 에이전트, 도구 호출 3-10회
- 직접 비교: 2-4 에이전트, 각 10-15회
- 복잡한 리서치: 10+ 에이전트, 책임 분할
- 위임 프롬프트엔 목적 + 출력 형식 + 도구 가이드 + 명확한 작업 경계 명시 (모호하면 중복·엇나감)

## 검증 — 환각 방지
- 에이전트의 "완료/검증됨" 선언을 그대로 믿지 말 것 — 실제 테스트 실행·diff 확인으로 교차검증
- 자동 검증 파이프라인도 실패한다(예: rate limit 으로 0건/전부실패) — 결과 형태가 의심스러우면 원인을 따진다

## 권한·보안 (자율 실행)
- 자율/auto 실행은 Docker 등 격리 환경에서, 되돌릴 수 있는 작업에 한해
- 안전 분류기도 오탐이 있다 — 고위험 작업은 인간 리뷰 대체 불가

## 안티패턴
- 거대 단일 CLAUDE.md 에 모든 규칙 욱여넣기 (준수율↓ 토큰↑)
- plan 없이 큰 기능 바로 구현
- 서브에이전트 장문 회신 → 컨텍스트 폭발
- 전 작업을 Opus 로 (Haiku/Sonnet 로 충분한 것까지)
- 오케스트레이션을 agent 로 설계 (깊이-1 로 작동 안 함 — skill 로)
